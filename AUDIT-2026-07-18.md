## 1. Резюме

Проект в хорошей инженерной форме: дисциплина транзакций, идемпотентность, fenced-обновления (lease/token_version), bounded-операции и интеграционные тесты на реальном PostgreSQL 17 — всё это на уровне выше среднего. С прошлого аудита закрыты обе reaper-задачи (#51/PR #52 — bounded reaping, PR #54 — capacity-доказательство на 100k).

Главные текущие риски — не в свежем коде, а в известных, но всё ещё открытых разрывах:

1. **TokenProvider держит блокирующую БД-транзакцию через внешний HTTP-вызов** — главный архитектурный риск доступности.
2. **`oauth_states` никогда не удаляются**, а `/oauth/start` — неаутентифицированный эндпоинт без rate limit: неограниченный рост таблицы + вектор флуда.
3. **Нет retention для `jobs`/`job_attempts`/`audit_log`/tombstones/workflow runs** — терминальные строки копятся вечно (признано в #21, не реализовано).
4. **Процессный сбой повторился в третий раз:** PR #54 влит, но чеклист #21 не отмечен, #12 всё ещё «pending external merge», контекстный issue #2 заморожен с 07-10.
5. **Нулевой контроль изменений:** branch protection на `main` отсутствует, все 15 влитых PR — 0 ревью.

## 2. Что хорошо (подтверждено чтением кода)

- **Очередь на PostgreSQL** ([internal/jobs/store.go](internal/jobs/store.go)): `SKIP LOCKED`, атомарный claim с инкрементом attempts, fenced `Complete`/`Fail`/`ExtendLease` (проверяют `locked_by`+`attempts`+`locked_until`), bounded reaping просроченных/исчерпанных задач с записью в `job_attempts` и observer-хуками.
- **Worker** ([internal/jobs/worker.go](internal/jobs/worker.go)): heartbeat на 1/3 lease с отменой контекста при потере lease, recover от паник хендлеров, финализация с некэнселируемым bounded-контекстом, drain с таймаутом.
- **Webhook ingress** ([internal/webhook/handler.go](internal/webhook/handler.go)): 2xx только после durable commit, глобальный + per-installation лимитеры (bounded, с TTL-евикцией), `MaxBytesReader`, сверка `account[id]` с installation, невалидные доставки сохраняются для аудита, а не отбрасываются молча.
- **Cleanup** ([internal/maintenance/cleanup.go](internal/maintenance/cleanup.go)): advisory-lock сериализует реплики, батчи с лимитом, `FOR UPDATE SKIP LOCKED`, метрики давления.
- **Секреты:** ключи шифрования versioned (AAD по installation/integration), refresh-токены и client secret зачищаются из памяти (`defer clear`), webhook key хранится хэшем, санитизация transport-ошибок (BUG-007).
- **Изоляция management-плоскости:** `/metrics` и `/ready` API вынесены на отдельный listener (PR #40).
- **Процессная база:** ADR-дисциплина, BUGS.md с regression-чеками, checkpoint-файлы, CI гоняет контракт, race-тесты и изолированные интеграционные тесты; guard на `migrate down` (требует `MIGRATION_DOWN_CONFIRM`).

## 3. Подтверждённые проблемы

### P1. TokenProvider держит row lock через внешний HTTP-вызов
[internal/oauth/token_provider.go:61](internal/oauth/token_provider.go:61) — `refreshIfVersion` открывает транзакцию, берёт `FOR UPDATE OF credentials` и **внутри неё** зовёт `p.gateway.Refresh(...)` (внешний HTTP к amoCRM, таймаут `AMOCRM_REQUEST_TIMEOUT=10s`). На время вызова заняты: соединение пула (при `DB_MAX_CONNS=10`) и row lock на credentials. Медленный/зависший amoCRM при нескольких параллельных refresh истощает пул и блокирует всё, что ходит в БД. Известный разрыв с прошлого аудита — не закрыт и не оформлен отдельным issue.

**Рекомендация:** перейти на паттерн «claim → HTTP вне транзакции → fenced finalize по `token_version`» (инфраструктура version-fencing уже есть); минимум — вынести в atomic issue под #11/#21 с явным capacity-риском.

### P2. `oauth_states` не очищаются, а `/oauth/start` не защищён
- В коде нет ни одного `DELETE FROM oauth_states`: строки создаются в [internal/oauth/store.go:153](internal/oauth/store.go:153) и помечаются consumed через UPDATE в [store.go:166](internal/oauth/store.go:166). Индекс `oauth_states_consumed_cleanup_idx` в схеме есть, но maintenance-пасс чистит только 4 таблицы (`used_widget_tokens`, `idempotency_keys`, `inbox_events`, `webhook_deliveries`) — таблица растёт неограниченно.
- `/oauth/start` и `/oauth/callback` ([cmd/api/main.go:142-143](cmd/api/main.go:142)) — публичные, без аутентификации и **без rate limiter** (лимитеры есть только у webhook ingress). Каждый запрос к start создаёт строку state → дешёвый вектор заполнения диска.

**Рекомендация:** добавить `oauth_states` в cleanup-политику (expired + consumed старше margin) и накрыть OAuth-эндпоинты хотя бы глобальным лимитером (механизм `IngressLimiter` уже готов).

### P3. Нет retention для jobs/job_attempts/audit_log/tombstones/workflow runs
Терминальные (`completed`/`dead`/`failed`) строки `jobs`, все `job_attempts`, `audit_log`, dedup-tombstones и workflow runs никогда не удаляются. Индекс `jobs_cleanup_idx` создан «на вырост», но политика не определена и код не написан — это признанный незакрытый пункт #21 («define finite retention for tombstones, workflow runs, outbound effects, jobs, and audit»). При заявленной цели 100k+ backlog деградация начнётся с планов запросов и bloat'а.

### P4. Post-merge sync снова не выполнен (третий повтор паттерна)
- PR #54 (100k capacity) влит 2026-07-12, но пункт «run production-like 100k expired-job capacity test» в чеклисте **#21 не отмечен**.
- Последний комментарий #12 — «Aggregate completion remains unchanged **pending external merge**», хотя merge давно состоялся.
- Контекстный issue #2 не обновлялся с 2026-07-10 (5 checkpoint-файлов и ~8 PR назад).
- Epic #6 (P5 jobs) не получил ссылок на #52/#54, хотя оба PR — про его подсистему.

Это ровно тот сбой, который ROADMAP.md сам описывает в «Протоколе после merge» (п. 1–5). Он повторяется на последнем PR каждой сессии: агент завершает работу до синхронизации.

**Рекомендация:** выполнить разовый sync (#21 отметить 100k-пункт со ссылкой на PR #54/CI, #12 обновить, #2 либо актуализировать, либо закрыть как замороженный в пользу CONTEXT.md). Системно — сделать sync частью acceptance самого PR (пункт в описании PR), а не пост-шагом сессии.

### P5. Нет контроля изменений на main
- Branch protection на `main` отсутствует (API отвечает 404 «Branch not protected»).
- Все 15 влитых PR имеют 0 ревью; required status checks не настроены — зелёный CI формально не обязателен для merge.

Для одиночного мейнтейнера это осознанный компромисс, но required CI checks стоят ноль усилий и защищают от случайного merge красной ветки.

### P6. Нет LICENSE
Репозиторий публичный, файла лицензии нет — по умолчанию «all rights reserved», что противоречит публичности и блокирует любое легальное переиспользование.

## 4. Возможные проблемы (умеренная уверенность)

- **Наблюдаемость API-процесса минимальна:** registry API регистрирует только Go/process-коллекторы и метрики ingress-лимитера ([cmd/api/main.go:61-65](cmd/api/main.go:61)); нет HTTP-метрик (латентность/коды по маршрутам) — SLO из #11 будет не на чем строить. У worker'а покрытие лучше (jobs, backlog gauges, cleanup, webhook).
- **Пропускная способность worker:** claim только по тику `PollInterval=1s`, батч 10, конкурентность 4 → устойчивый потолок ~10 задач/сек на реплику при одной допустимой реплике (ADR). Для целевых нагрузок стоит замерить и зафиксировать в capacity-доке; сами reaper'ы уже доказаны на 100k.
- **Dev-ключ шифрования в `.env.example`** (`1:MDEyMzQ1...` = «0123456789abcdef…»): валидный по формату — риск, что попадёт в прод копипастой. Дешёвая защита: в `production` окружении отказываться стартовать с этим известным ключом.
- **`SaveInvalidDelivery` пишет тело до какой-либо валидации содержимого** — смягчено лимитерами, `MaxBytesReader` и retention 720h, но осознанно принятая стоимость диска; при злоупотреблении конкретным webhook key затопить можно только в пределах rate limit.
- **`ExtendLease` требует `locked_until >= now()`:** после паузы процесса (GC/суспенд контейнера) дольше остатка lease задача корректно теряется и переиграется — поведение правильное, но означает, что `WORKER_LEASE_DURATION=1m` нельзя сильно уменьшать.
